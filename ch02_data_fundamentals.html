Data Fundamentals — Types, Quality, and Structure

Before building models, running experiments, or drawing conclusions, you need to understand what your data actually is — its type, its structure, and where it can go wrong. This chapter covers the vocabulary that appears in every data science brief, every model specification, and every analytical QA conversation.

Terms covered: 24 Sections: Variable types · Data quality · Missing data · Leakage & drift Audience: Strategy, PM, Analytics-adjacent
Section 2.1

Variable types and measurement scales

Not all data is the same kind of thing. Whether a variable is a category, a rank, a count, or a continuous measurement determines which statistical methods are valid, how it should be encoded for models, and how it should be displayed on dashboards. Using the wrong method for the wrong data type is one of the most common sources of silent analytical errors.

The distinction between types is not academic. A model that treats a nominal category like region as if it were an ordered number will invent a fake ordering — North (1) is not "less than" South (2). A correlation calculation applied to ordinal survey responses assumes equal gaps between "agree" and "strongly agree" that do not exist. Getting variable types right is the prerequisite for everything else.

Nominal
Categorical
Discrete groups with no inherent order. Averaging is meaningless.
region, product_type, channel, payment_method
Ordinal
Ordered Categorical
Categories with a meaningful order but unknown gap sizes between levels.
satisfaction (low/med/high), risk_band, loyalty_tier
Binary
Binary / Indicator
Only two values. Technically categorical but treated specially in models.
churned (0/1), has_prior_claim, email_opened
Discrete
Count / Integer
Numeric, restricted to whole numbers. Cannot be negative in most business contexts.
order_count, support_calls, num_products_held
Continuous
Continuous
Any numeric value in a range. Differences and ratios are both meaningful.
revenue, age, delivery_time_hours, cltv
Temporal
Date / Time Series
Observations indexed by time. Order matters — shuffling destroys information.
daily_sales, login_timestamp, cohort_week
The silent error to watch for Categorical variables are frequently stored as integers in databases — region coded as 1, 2, 3, or product tier coded as 1, 2, 3, 4. If these get fed into a model as numeric values without encoding, the model infers an ordering and spacing that does not exist. Always check the data dictionary before assuming an integer column is a true numeric variable.
Categorical Variable
A variable that places observations into distinct groups with no inherent numeric meaning.
expand
Intuition
It tells you which bucket an observation belongs to — not how much of something it has. You cannot average a categorical variable. You can count frequencies, compare proportions, and test whether distributions differ across groups.
Formula / Rule
Nominal: treat as labels only — no arithmetic. For modeling, encode using one-hot encoding (one binary column per category) or target encoding (replace with outcome average). Never feed raw category codes as numbers into a linear model.
When you see it
In customer segmentation tables, product mix reports, survey response data, and model feature specifications where column values are labels rather than quantities.
Red flag
Ask whether integer-coded categories were accidentally treated as numeric in any preprocessing step. A "mean region" is a meaningless number that a model will happily learn from if not caught.
Business example
A retail bank's churn model includes account_type coded as 1 (checking), 2 (savings), 3 (investment). Without one-hot encoding, the model learns that investment accounts are "3 times" checking accounts — a fabricated relationship that distorts coefficient estimates for every variable correlated with account type.
Ordinal Variable
A categorical variable whose levels have a meaningful order but unknown gap sizes between them.
expand
Intuition
It tells you observations can be ranked but not measured. The difference between "satisfied" and "very satisfied" is not the same as the difference between "dissatisfied" and "satisfied" — or at least there is no evidence it is. Treating ordinal data as if gaps are equal systematically distorts averages and model coefficients.
Formula / Rule
Use the rank information — avoid computing arithmetic means of ordinal codes. For 5-point Likert scales, report medians or distributions rather than means. For modeling, either treat as ordered categories or apply label encoding if tree-based models will handle the nonlinearity.
When you see it
In customer satisfaction surveys (NPS detractor/passive/promoter), employee engagement scores, credit risk bands, SLA priority levels, and any five-point Likert scale.
Red flag
Ask whether the analysis is computing a mean of ordinal codes and presenting it as if it were a continuous measurement. "Average satisfaction of 3.8 out of 5" assumes equal gaps that the scale does not guarantee.
Business example
A healthcare insurer tracks member satisfaction on a five-point scale. The analytics team reports "average satisfaction improved from 3.4 to 3.7." Whether this is meaningful depends on whether the middle of the scale represents a psychologically equal distance from each adjacent point — an assumption that survey research frequently violates.
Continuous Variable
A numeric variable that can take any value in a range, where differences and ratios are both meaningful.
expand
Intuition
It tells you the metric is measured on a true numeric scale where arithmetic is valid. You can compute means, standard deviations, correlations, and regression coefficients. Most quantitative analytical methods are designed for continuous variables, which is why so much effort goes into making other variable types behave like them.
Formula / Rule
Check two things before treating a variable as continuous: (1) can it take non-integer values, and (2) does zero have a meaningful interpretation? Revenue has a true zero; temperature in Celsius does not — which affects which statistics make sense.
When you see it
In financial metrics, time measurements, behavioral counts, pricing data, and most model input and output variables.
Red flag
Ask whether the variable can plausibly take the values implied by treating it as continuous. Order count is technically discrete — and models trained on it may predict negative orders unless the structure is acknowledged.
Business example
An e-commerce company models delivery time as a continuous variable in days. This is appropriate — 1.5 days is a meaningful value, the scale has a true zero, and arithmetic comparisons between values carry real meaning for customer experience planning.
Binary Variable
A variable with exactly two values, usually coded 0 and 1, representing presence or absence of a condition.
expand
Intuition
It tells you whether something happened or a condition holds. Binary variables are the targets for classification models (churned or not), the outputs of rule-based flags, and common inputs to regression when a categorical variable has only two levels. The mean of a binary variable is the proportion of 1s — one of the few cases where averaging a non-continuous variable is perfectly valid.
Formula / Rule
Mean of binary variable = proportion of 1s = P(event) A binary variable with mean 0.04 means the event occurs in 4% of observations. That proportion is the base rate — a critical input for understanding model performance in rare-event settings.
When you see it
As model targets (default/no default, churn/retain, fraud/not fraud), as flags in customer data (has_complaint, is_loyalty_member), and as encoded dummy variables in regression.
Red flag
Ask what coding convention was used and whether it is consistent. If 1 means churned in one dataset and 1 means retained in another, joining them creates a perfectly inverted target variable — a catastrophic and silent error.
Business example
A lender's model uses a binary default indicator as its target. The variable has a mean of 0.032 — meaning only 3.2 percent of the population defaulted. This low base rate means accuracy is a misleading evaluation metric; a model predicting "no default" for everyone would be 96.8 percent accurate while being completely useless.
Structured vs Unstructured Data
Structured data fits predefined rows and columns; unstructured data is free-form content — text, images, audio, or documents.
expand
Intuition
It tells you whether the data is analysis-ready or requires a transformation step first. Structured data — transaction records, CRM fields, clickstream logs — can be queried and modeled directly. Unstructured data — call center transcripts, support ticket text, contract documents — contains often richer signal but requires NLP, OCR, or feature extraction before it can be used analytically.
Formula / Rule
Structured: rows × columns table with a defined schema. Unstructured: raw content requiring preprocessing to extract features. Semi-structured (JSON logs, API responses) sits between — it has some organization but no rigid schema.
When you see it
In data strategy discussions, analytics scoping, AI use-case prioritization, and any conversation about what data "we have" versus what data "we can analyze."
Red flag
Ask whether "available data" means structured and analysis-ready or merely stored somewhere. An insurer may have years of adjuster notes — rich unstructured signal — but if those have never been processed into features, they are not available for a model without significant additional work.
Business example
A telecom operator has structured billing records, usage logs, and network events alongside unstructured customer service call transcripts. The structured data feeds churn and revenue models directly. Extracting complaint themes and sentiment from the transcripts requires an NLP pipeline first — adding weeks to a project timeline and a separate governance consideration.
Section 2.2

Data quality — the silent driver of analytical failure

Most analytics projects fail or underperform not because of algorithm choice or model complexity, but because of data quality problems that were not caught before analysis began. Poor quality data does not announce itself — it produces results that look plausible, pass initial checks, and only reveal themselves when predictions fail in production or decisions go wrong at scale.

Data quality has four dimensions worth keeping distinct. Completeness — are the values present? Accuracy — do the values reflect reality? Consistency — do the same entities look the same across different sources and time periods? Timeliness — was the data current at the moment the prediction needed to be made? Each can fail independently, and each failure has different consequences and remedies.

The four dimensions of data quality — each can fail independently and each creates different downstream errors.

Duplicates Same transaction appears twice; customer_id matches multiple records Inflated counts, double-weighted observations distort model training Count distinct vs total rows; join cardinality checks

Outlier Treatment
The handling of unusually extreme observations — through removal, capping, transformation, or separate modeling.
expand
Intuition
It tells you the team made a deliberate choice about extreme values rather than letting them distort results silently. The choice matters enormously: removing an outlier that is actually the most commercially important case is a much worse error than keeping it.
Formula / Rule
IQR rule: flag values below Q1 − 1.5×IQR or above Q3 + 1.5×IQR Common approaches: (1) remove — only if confirmed data error. (2) Winsorize — cap at a chosen percentile (e.g. 99th) to limit influence while preserving the observation. (3) Transform — log or square root to compress the scale. (4) Separate model — model extremes explicitly if they follow different patterns.
When you see it
In data cleaning pipelines, preprocessing decisions, model documentation, and post-hoc reviews when a model underperforms on high-value customers or edge cases.
Red flag
Ask whether the outlier is a data error or a real business case. A payment of £450,000 may be outlying in distribution but represent the single most strategically important transaction in the dataset — capping it silently destroys value signal.
Business example
A payments company caps transaction amounts at the 99th percentile before training a fraud model. The top 1 percent of transactions are the highest-value targets for fraud — by treating them as noise, the model is systematically weaker exactly where it matters most. The cap should at minimum be documented and the model's performance on those transactions tested separately.
Imputation
The process of filling in missing values using a chosen rule or model estimate.
expand
Intuition
It tells you the team replaced gaps in the data so analysis or modeling could proceed. Imputation is always an assumption — it substitutes a manufactured value for an unknown one. The method chosen should match the reason values are missing, not just the statistical convenience of the analyst.
Formula / Rule
Mean/median imputation: replace with column average or median. Fast, but reduces variance and distorts correlations. Regression imputation: predict missing values from other columns. KNN imputation: fill from similar complete records. Indicator variable: always add a binary flag for whether the value was missing — the pattern of missingness is itself often predictive.
When you see it
In preprocessing pipelines, feature engineering documentation, model cards, and any analysis where the raw data had non-trivial null rates.
Red flag
Ask whether a missingness indicator was also created alongside the imputed value. A customer with missing income is different from one with high reported income — the missingness itself may carry predictive signal that gets discarded by silent imputation.
Business example
A mortgage lender imputes missing self-employment income with the column median. Self-employed applicants have higher income variance and structurally different repayment patterns — median imputation understates their actual income for high earners and makes the model treat them as near-average when they are categorically different. A missingness flag plus segment-specific imputation would be materially more accurate.
Section 2.3

Missing data mechanisms — why it matters how values go missing

Not all missing data is the same. The reason values are absent determines whether you can safely impute, whether you need to model the missingness itself, and whether any analysis using that column will be biased. This is one of the most underappreciated distinctions in applied analytics — and the one most commonly ignored in practice.

MCAR
Missing Completely at Random
Values are absent for reasons unrelated to any observed or unobserved data. The easiest case — imputation introduces no systematic bias.
Example: survey respondents randomly skipped a question due to form rendering errors on certain device types.
MAR
Missing at Random
Missingness is related to observed data but not to the missing value itself. Can be addressed by conditioning on other variables — more manageable than MNAR.
Example: older customers are less likely to provide email addresses — missingness depends on age (observed), not on the email address value itself.
MNAR
Missing Not at Random
Missingness is related to the missing value itself. The hardest case — naive imputation introduces systematic bias that is difficult to correct.
Example: high earners decline to report income. The missing income values are systematically higher than reported ones — imputing with the mean understates income for the most important segment.
Why the mechanism matters in practice If data is MNAR and you impute with the mean, you are replacing high values with average values — the analysis will systematically underestimate effects for the population where missingness is concentrated. In credit models, this can mean systematically mispricing risk for the most creditworthy or least creditworthy segments. Always ask: could the type of person who doesn't answer this question be systematically different from those who do?
Missing Data
Values that are absent from the dataset for some observations — requiring a decision about how to handle them before analysis proceeds.
expand
Intuition
It tells you the data is incomplete in a way that may or may not be random. Missing data is not just a technical problem to solve before modeling — it is information about the data-generating process. The fact that something is missing can be as telling as what value was recorded.
Formula / Rule
Step 1: quantify missingness rate per column. Step 2: check whether missingness correlates with the outcome or key predictors. Step 3: determine the likely mechanism (MCAR / MAR / MNAR). Step 4: choose imputation method appropriate to the mechanism. Step 5: always create a missingness indicator flag alongside any imputed value.
When you see it
In every real-world dataset — missing rates above 5 percent on key variables are worth investigating. Above 20 percent should trigger a question about whether the variable is usable at all.
Red flag
Ask why the data is missing and whether the pattern of missingness correlates with the outcome. If it does, the missingness is itself a signal — and treating it as a nuisance to be filled in silently will destroy that signal.
Business example
A fintech lender's model uses a credit bureau score that is absent for 18 percent of applicants — primarily recent immigrants and young adults with no credit history. These applicants are not missing at random; they represent a distinct segment. Imputing their score with the population median misclassifies an underserved population that may actually be a strong credit risk based on alternative signals.
Section 2.4

Data leakage, look-ahead bias, and drift — three ways data silently breaks models

These three concepts are the most consequential data quality issues for predictive models. Unlike missing values or type errors, they do not produce obvious errors during development — they produce models that look excellent and then fail quietly or catastrophically in production. Every model deployment should be checked for all three before going live.

Data Leakage
Using information during model training that would not actually be available at the moment the model needs to make a prediction.
expand
Intuition
It tells you the model is cheating — often without anyone knowing it. Leakage creates false confidence: training performance looks exceptional, validation metrics are strong, and the model is deployed. Then it fails immediately in production because the features that drove its training-time performance are not available when real predictions are needed.
Formula / Rule
For every feature ask: would this value be known at the exact moment this prediction needs to be made in production? If a feature is derived from the outcome, computed after the event, or available only in hindsight — it is leakage. AUC of 0.99+ on a real business problem almost always signals leakage.
When you see it
In model development reviews, production readiness checks, and post-deployment diagnosis when model performance drops sharply from development metrics.
Red flag
Ask whether performance metrics are suspiciously high. On real, messy business data, an AUC above 0.95 for a churn or default model should trigger immediate leakage investigation — real business signals rarely separate that cleanly.
Business example
A hospital readmission model includes "total cost of care in the 30 days following discharge" as a feature. That variable is determined after the prediction event — a patient cannot be scored at discharge using information from their subsequent stay. The model achieves 0.97 AUC in development and near-random performance in production. The leaky feature drove all the apparent predictive power.
Target Leakage
A specific form of leakage where a feature is an effect or proxy of the target variable — making prediction artificially easy and useless in practice.
expand
Intuition
It tells you a feature was included that essentially gives the answer away — not because it predicts the future, but because it reflects the outcome that has already happened. The model learns from a label disguised as a feature. This is distinct from general leakage: target leakage specifically involves including outcomes or their immediate proxies as inputs.
Formula / Rule
Ask for every high-importance feature: could this variable have been computed from or after the target? If yes, it is target leakage. Common examples: including cancellation_reason to predict churn (the reason is recorded at cancellation), or including claim_status_final to predict fraud (the status reflects the investigation outcome).
When you see it
In feature importance reports where one variable dominates all others by a wide margin, in models where SHAP values point to a single feature accounting for most predictions.
Red flag
Ask what the top SHAP feature represents and trace its data lineage. If a feature that should not logically be the most powerful driver is shown as far more important than business experts would expect, investigate its construction immediately.
Business example
A subscription churn model includes "days_since_last_login_at_cancellation_date" as a feature. That variable is computed on the same day the customer cancels — it is an artefact of the outcome, not an early warning signal. The model achieves near-perfect training performance and zero production value.
Look-Ahead Bias
Using future information in a historical analysis — making backtests or validation results look better than any real-world deployment could achieve.
expand
Intuition
It tells you the model was trained or evaluated with hindsight it would never have in production. Look-ahead bias is especially common in time series work — because time-based features are easy to miscalculate — and in financial backtesting, where it can make worthless strategies appear profitable.
Formula / Rule
For time series: every feature must use only data available up to and including the prediction date. Reporting lag matters — a financial figure published 30 days after month-end cannot be used as a feature for a model predicting events on month-end. Always use point-in-time feature construction.
When you see it
In time-series forecasting, financial model backtesting, demand planning models, and any analysis where the same data is used to construct both features and labels for overlapping time windows.
Red flag
Ask when each feature value was actually available versus when the event being predicted occurred. A 1-day lag assumption may be wrong if data pipelines have 3-day refresh cycles — the feature would not have been current at prediction time.
Business example
An asset manager backtests a strategy using end-of-month accounting data. The data is published 25 days after month-end. When the backtest uses the data as if it were available on month-end, the strategy appears profitable. When the backtest is reconstructed using publication dates, performance is materially worse — the look-ahead bias was worth 6 percentage points of annual return.
Data Drift
A change over time in the distribution of input features used by a model — signaling that the population reaching the model is no longer the same as when it was trained.
expand
Intuition
It tells you the model's inputs are shifting in ways it was not designed for. A model trained on last year's customer mix may receive a materially different customer mix this year — and the features that were predictive then may no longer carry the same signal. Data drift can degrade performance even when the underlying relationship between inputs and outcomes has not changed.
Formula / Rule
PSI = Σ (Actual% − Expected%) × ln(Actual% / Expected%) Population Stability Index (PSI): compare current feature distributions with training-time distributions. PSI below 0.1 = stable. PSI 0.1–0.2 = moderate drift, investigate. PSI above 0.2 = significant drift, revalidate model performance.
When you see it
In model monitoring dashboards, post-deployment performance reviews, and any context where the customer mix, product mix, or channel composition has changed significantly since the model was trained.
Red flag
Ask which variables have drifted and whether those variables are important predictors in the model. Drift in an irrelevant feature is less concerning than drift in a top-5 SHAP feature — focus investigation on the features that drive model decisions.
Business example
A lender's credit model was trained on applications acquired primarily through branch channels. After launching a digital acquisition programme, the applicant mix shifts: younger, thinner credit files, different income profiles. PSI spikes on income and employment tenure features. The model's Gini drops 8 points over the next quarter — data drift preceded the performance decay by three months.
Concept Drift
A change over time in the fundamental relationship between inputs and the target outcome — making a model's learned patterns obsolete even if the inputs themselves are stable.
expand
Intuition
It tells you the world has changed in a way that invalidates what the model learned. Data drift affects the inputs; concept drift affects the relationship between inputs and outputs. Concept drift is harder to detect because input monitoring looks normal — only outcome monitoring reveals that predictions are becoming systematically wrong.
Formula / Rule
Monitor: (1) model calibration over time — are predicted probabilities still matching observed rates? (2) Model discrimination — is Gini or AUC holding? (3) Feature-outcome correlations — are top predictors still as strongly associated with the outcome? If (1) and (2) deteriorate while PSI is stable, concept drift is the likely cause.
When you see it
After major market or behavioral shifts: COVID-19, a pricing restructure, a regulatory change, a major competitor entering the market. Any event that genuinely changes how customers behave relative to what was historically predictive.
Red flag
Ask whether performance decay is happening even though input distributions look stable. If PSI is low but model accuracy is falling, concept drift is the likely culprit — and more data in the old pattern will not fix it. A retrain on recent data is needed.
Business example
A retail churn model trained pre-pandemic used in-store visit frequency as a top predictor. After lockdowns, customers who previously visited often now visit rarely but have not churned — they shifted to online. The old relationship between visit frequency and churn is inverted. Adding more pre-pandemic data makes the model worse; it needs to be retrained on post-behavioral-shift data.
PSI — Population Stability Index
A metric that quantifies how much a variable's distribution has shifted from a reference period — the standard early warning tool for model input monitoring.
expand
Intuition
It tells you whether the population reaching the model still looks like the population it was trained on. PSI is not proof that the model has deteriorated — it is a trigger to investigate whether deterioration may be occurring. A high PSI on a weak predictor matters less than a moderate PSI on a top-5 predictor.
Formula / Rule
PSI = Σ (A_i − E_i) × ln(A_i / E_i) Where A_i = actual proportion in current period for bin i, E_i = expected proportion from training period. Bin the variable into 10 equal buckets. PSI < 0.1: stable. PSI 0.1–0.2: monitor closely. PSI > 0.2: investigate and consider revalidation.
When you see it
In model monitoring reports, credit governance packs, scorecard validation tables, and any production analytics environment where models are tracked over time.
Red flag
Ask what action follows a PSI alert and whether the investigation has been completed. PSI is an alarm — not a diagnosis. Seeing a flagged PSI in a governance report with no accompanying investigation note is a model risk management gap.
Business example
A card issuer monitors PSI monthly across 40 model input features after launching a new digital acquisition channel. Three features — age, employment type, and revolving utilization — show PSI above 0.2 in month four. All three are top-10 SHAP predictors. The governance team triggers an out-of-cycle model review, finding the model's Gini has already deteriorated by 6 points.
Chapter Summary

Chapter 2 cheat sheet

Key principles from this chapter
Variable type determines valid operations
Categorical cannot be averaged. Ordinal gaps are not equal. Integer-coded categories are not numeric. Check types before any operation.
Data quality has four dimensions
Completeness, accuracy, consistency, and timeliness can fail independently. Each failure has different consequences and different remedies.
Why values are missing matters as much as that they are missing
MCAR allows simple imputation. MAR allows conditional imputation. MNAR requires modeling the missingness mechanism — silent mean imputation creates systematic bias.
Leakage makes models look better than they are
Data leakage, target leakage, and look-ahead bias all produce optimistic development metrics and poor production performance. AUC above 0.95 on real data should always trigger leakage investigation.
Drift comes in two forms with different fixes
Data drift: input distributions shift — monitored by PSI. Concept drift: input-outcome relationship shifts — only revealed by outcome monitoring. More training data fixes neither if the world has changed.
PSI is a trigger, not a verdict
PSI above 0.2 means investigate, not retire. Prioritise investigation for features that are top predictors — not all drift matters equally.
