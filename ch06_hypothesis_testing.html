Hypothesis Testing — Is This Real or Just Noise?

Hypothesis testing is the formal framework for deciding whether observed patterns in data are real enough to act on. It is also one of the most widely misunderstood and misused tools in business analytics. This chapter makes you a harder, more discerning audience for significance claims.

Terms covered: 19 Sections: Framework · p-values · Error types · Power · Multiple testing · Pitfalls
Section 6.1

The hypothesis testing framework

Hypothesis testing starts with a default assumption (null hypothesis) and asks whether the data provides strong enough evidence to reject it. It is a courtroom logic: the null hypothesis is innocent until proven guilty beyond a reasonable doubt — where "reasonable doubt" is defined by the chosen significance level. Understanding this framework is essential for interpreting experiment results, regression outputs, and any claim that something "reached significance."

The null distribution shows what test statistics look like when H₀ is true. The p-value is the probability of seeing a result at least as extreme as the observed one — the red shaded area. If the observed statistic falls in the critical region, we reject H₀.
Null Hypothesis (H₀)
The default claim — no effect, no difference, no relationship — which the data must provide strong evidence against before it is rejected.
expand
Intuition
It tells you what the test treats as the status quo. The burden of proof lies entirely with the data — the null is assumed true unless the evidence against it is overwhelming. In business terms: "this change did not improve the metric" is the null. The test asks if the data provides enough evidence to reject that default assumption and declare the change effective.
Formula / Rule
H₀: μ₁ = μ₂ (or: effect = 0, or: β = 0)The null specifies the value of the parameter under "no effect." Common forms: means are equal, proportion difference is zero, regression coefficient is zero, correlation is zero. The null should always be defined before seeing the data.
When you see it
In A/B test setup, regression output (each coefficient has an associated H₀: β = 0), chi-squared tests, and any formal statistical test report.
Red flag
Ask whether the null hypothesis reflects the real business question. The null of "no difference in means" may not be the same as "not worth rolling out" — a tiny positive effect that passes significance testing may still be too small to justify the cost of implementation.
Business example
A streaming service tests a new recommendation algorithm. H₀: daily watch time per user is the same under both algorithms. After one week, the new algorithm shows +4.2 minutes per user per day (p = 0.003). The test rejects the null — but the team also checks whether 4 extra minutes justifies the engineering cost and latency increase of the new algorithm.
P-Value
The probability of observing results at least as extreme as those obtained, assuming the null hypothesis is true.
expand
Intuition
It tells you how surprising the observed data is under the assumption that nothing is happening. A small p-value means "if the null were true, this result or something more extreme would occur very rarely" — which is evidence against the null. It does not tell you the probability that the null is true, the probability that the alternative is true, or how large the effect is.
Formula / Rule
p-value = P(data as extreme or more | H₀ true)Compute the test statistic from the data. Ask: if H₀ were true, what fraction of possible samples would produce this extreme or a more extreme test statistic? That fraction is the p-value. Decision: reject H₀ if p-value < α (significance level). Never: accept H₀ if p-value > α — you merely "fail to reject."
When you see it
In A/B test dashboards, regression outputs (p-values on each coefficient), model comparison tests, and any table reporting statistical test results.
Red flag
Ask for the effect size alongside the p-value. With a large enough sample, any non-zero effect becomes statistically significant regardless of practical importance. A p-value of 0.0001 on a 0.02% conversion lift is statistically significant and commercially irrelevant.
Business example
A retailer with 500,000 weekly website visitors tests a button color change. With this traffic volume, even a 0.01% conversion difference produces p < 0.001. The result is "highly significant" by any conventional threshold — but 0.01% lift on 500,000 visitors generates ~50 extra transactions per week. Whether that justifies even a half-day of engineering work requires calculating expected revenue, not reading a p-value.
Significance Level (α)
The pre-chosen threshold for rejecting the null hypothesis — the acceptable false positive rate.
expand
Intuition
It tells you how strict the test is about false alarms. Setting α = 0.05 means: if the null is true, I will incorrectly reject it 5% of the time — and I am comfortable accepting that error rate. Lower α (0.01, 0.001) means requiring stronger evidence before declaring a result, at the cost of missing more real effects.
Formula / Rule
Reject H₀ ⟺ p-value < αα must be set before the analysis. Conventional defaults: α = 0.05 (general business), α = 0.01 (higher stakes), α = 0.001 (safety-critical). The choice should reflect the cost of a false positive — launching a bad feature, publishing wrong findings, making a false medical claim. Never adjust α after seeing the data.
When you see it
In experiment pre-registration, test design, methodology sections, and any analysis where a binary "significant/not significant" decision is made.
Red flag
Ask whether α was set before the analysis. Choosing α = 0.05 and finding p = 0.04 is different from planning to use α = 0.01, finding p = 0.04, then switching to α = 0.05 to achieve significance. The latter inflates false positive rates.
Business example
A medical device company sets α = 0.001 for a clinical trial — much stricter than the default 0.05 used for a marketing campaign test. The asymmetry is deliberate: a false positive in device safety leads to patient harm; a false positive in marketing wastes budget. Business context should always determine significance level, not convention.
Section 6.2

Type I and Type II errors — the fundamental tradeoff

Every hypothesis test can be wrong in two different ways. Understanding which type of error is more costly for a given business decision is the key to setting up tests correctly — and to interpreting their results appropriately.

Type I Error — False Positive
Rejecting H₀ when it is true
Declaring an effect exists when it doesn't. Probability = α (the significance level you chose). You controlled this when setting up the test.
Business cost: rolling out a feature that does nothing. Wasted engineering, distracted roadmap, misplaced confidence.
Type II Error — False Negative
Failing to reject H₀ when it is false
Missing a real effect. Probability = β (related to power: 1−β = power). Reducing Type II error requires larger samples or higher α.
Business cost: failing to ship a genuinely better feature. Lost competitive improvement, missed opportunity.
The error tradeoff in practice Reducing α (stricter threshold) reduces Type I errors but increases Type II errors — you miss more real effects. Increasing sample size reduces both error types simultaneously — the only free lunch in hypothesis testing. For decisions where false positives are cheap and false negatives are costly (e.g., testing a new onboarding flow), relax α or increase power. For decisions where false positives are costly (e.g., medical claims, safety decisions), tighten α.
Type I Error (False Positive)
Rejecting the null hypothesis when it is actually true — declaring a win when nothing changed.
expand
Intuition
It tells you the test produced a false alarm. At α = 0.05, this happens 5% of the time even when the null is true — meaning if you run 20 independent tests on truly ineffective changes, you expect one false positive purely by chance. This is not a flaw in the test — it is a direct consequence of the acceptable error rate you chose.
Formula / Rule
P(Type I Error) = αYou directly control Type I error rate via the significance level. But control applies on average, not to any specific test. The test cannot tell you whether this particular significant result is a true positive or the 1-in-20 false alarm.
When you see it
In discussions about what false positive rate is acceptable for the decision at hand, and in retrospective analysis when "significant" A/B test winners fail to produce expected gains post-launch.
Red flag
Ask what the business cost is if this statistically significant result is actually a Type I error. A significant result that would cost millions to implement should require stronger evidence (lower α) than a result requiring only an afternoon of engineering work.
Business example
A fintech product team runs 30 A/B tests per quarter at α = 0.05 on genuinely null changes (no real effects). They expect approximately 1.5 false positives per quarter — tests that will appear significant purely by chance. Without pre-registration and holdback validation, these false wins enter the product roadmap and fail to replicate post-launch.
Type II Error (False Negative)
Failing to reject the null hypothesis when it is false — missing a real effect.
expand
Intuition
It tells you the test was not sensitive enough to detect a real effect — the improvement was genuine, but the data was too noisy or too small to confirm it. The most common cause is running tests with insufficient sample sizes. A failed test that was underpowered cannot be interpreted as evidence the null is true — it merely tells you the test was too weak to detect the effect.
Formula / Rule
P(Type II Error) = β = 1 − PowerYou control β through sample size and effect size specification. To reduce Type II errors: collect more data, lower the noise (better measurement, more homogeneous population), or accept detecting only larger effects. Underpowered tests (β > 0.2, power < 80%) should not be run — their non-significant results are nearly uninterpretable.
When you see it
In sample size planning, power analysis, and post-test evaluation when results are non-significant — particularly when the team expected a positive effect but the data "didn't show it."
Red flag
Ask whether the test was adequately powered before interpreting a null result as evidence of no effect. "We ran the test and found no significant improvement" is meaningless without knowing whether the test had 80% power to detect a commercially meaningful effect.
Business example
A SaaS company tests a new onboarding flow on 400 users per arm, expecting a 15% improvement in 30-day activation. The test returns p = 0.14 — not significant at α = 0.05. But power analysis shows the test was only 45% powered to detect a 15% improvement at this sample size. The result is ambiguous — not "the flow doesn't work" but "we can't tell with this much data."
Statistical Power
The probability of correctly detecting a real effect of a given size — the test's sensitivity.
expand
Intuition
It tells you whether the test has a fair chance of finding what you're looking for. Low power means you will often miss genuine effects — not because they are not there, but because the data is too noisy or sparse to detect them reliably. The industry standard of 80% power means accepting a 20% chance of missing a real effect of the specified size.
Formula / Rule
Power = 1 − β = P(reject H₀ | H₁ true)Power increases with: larger sample size, larger true effect size, lower variability, and higher α. Required sample size: n ≈ (z_α/2 + z_β)² × 2σ² / δ² for two-sample mean comparison. Target ≥80% power (β ≤ 0.20). Use 90% power for high-stakes decisions.
When you see it
In experiment pre-planning, sample size calculators, power analysis reports, and any discussion of whether an experiment was adequately designed before interpreting results.
Red flag
Ask what minimum detectable effect the experiment was designed to detect. A non-significant result from a test designed to detect 5% lift tells you almost nothing about whether there is a 2% lift — the test was never set up to see that.
Business example
A grocery retailer wants to detect a 2% uplift in basket size from a loyalty programme pilot. With weekly basket SD of £18 and α = 0.05, achieving 80% power requires approximately 3,140 customers per arm. Running the pilot on 500 per arm would have only 28% power — almost certain to miss a real 2% effect. The decision to run the pilot at all should depend on whether 6,280 customers can be enrolled.
Effect Size and Practical Significance
The magnitude of a difference or relationship, independent of sample size — the "how much" beyond the "is it real."
expand
Intuition
It tells you whether a statistically significant result is also commercially meaningful. Statistical significance tells you the effect likely exists. Effect size tells you whether it is large enough to matter. With large samples, even trivially small effects reach significance — which is why every significant result should be accompanied by an effect size and a business value calculation.
Formula / Rule
Cohen's d = (μ₁ − μ₂) / s_pooledSmall: d = 0.2. Medium: d = 0.5. Large: d = 0.8. For business purposes: state the effect in business units first (£ revenue change, % conversion lift, minutes saved) before any standardised metric. Business stakeholders care whether the effect justifies the cost — Cohen's d is a secondary diagnostic.
When you see it
In experiment result write-ups, meta-analyses, power analyses, and any context where distinguishing statistical significance from business importance matters.
Red flag
Ask for the effect in business units alongside the p-value. A p-value of 0.001 with an effect of £0.03 per customer per year says this test should not be on the roadmap, regardless of its statistical purity.
Business example
An e-commerce platform's A/B test finds a button placement change that increases conversion rate by 0.08 percentage points (from 3.42% to 3.50%). With 800,000 monthly visitors, this is 640 extra conversions at £45 AOV = £28,800/month. Statistically significant (p < 0.001) and commercially worthwhile (£346k/year with minimal maintenance cost). Effect size puts the significance in business context.
Section 6.3

Multiple testing — the silent false positive factory

The multiple testing problem is one of the most underappreciated sources of spurious findings in business analytics. Every additional test you run at α = 0.05 creates another 5% chance of a false positive from noise alone. Run 20 tests and you expect one false positive even if nothing is real. Run 100 tests and you expect five. This is not a failure of the tests — it is probability math.

Multiple Testing Problem
The inflated false positive rate that results from testing many hypotheses simultaneously without adjusting for the number of comparisons.
expand
Intuition
It tells you that the more things you test, the more "significant" results you will find by chance — even if all the null hypotheses are true. This is not cheating or bad faith; it is basic probability. With 20 independent tests at α = 0.05: P(at least one false positive) = 1 − 0.95²⁰ ≈ 64%. More than half the time, you will find at least one false positive regardless of whether anything real happened.
Formula / Rule
P(≥1 false positive | m tests, all H₀ true) = 1 − (1−α)ᵐFor m = 20 tests, α = 0.05: probability ≈ 0.64. This is the Family-Wise Error Rate (FWER) — the probability of at least one false positive across the family of tests. Solutions: Bonferroni correction, Benjamini-Hochberg FDR control, or pre-registering only the primary test hypothesis.
When you see it
In dashboards with many KPIs, A/B tests with many subgroup cuts, feature screening in ML, and any analytical context where many tests are run and results cherry-picked.
Red flag
Ask how many tests were run to arrive at this significant result. If someone says "we found a significant lift in the 35–44 female urban segment" after testing 40 demographic subgroups, that result needs a Bonferroni correction or a replication before being trusted.
Business example
A subscription platform launches a feature and then checks impact across 8 user segments, 12 engagement metrics, and 3 time windows — 288 combinations in total. Reporting only the 15 combinations that show p < 0.05 as "findings" is statistically indistinguishable from random noise. Expected false positives at α = 0.05: approximately 14. Almost all 15 "findings" are likely spurious.
Bonferroni Correction
Dividing the significance threshold by the number of tests to control family-wise error rate — conservative but simple.
expand
Intuition
It tells you the test is being held to a stricter standard because many comparisons are being made. If you run 20 tests and want to keep overall false positive rate at 5%, each individual test needs a threshold of 0.05/20 = 0.0025. It is simple, transparent, and conservative — it may be too strict when tests are correlated, making it harder to detect real effects.
Formula / Rule
α_adjusted = α / m (where m = number of tests)Controls FWER at α. Conservative because it assumes tests are independent — correlated tests need less correction. For large m, consider FDR control (Benjamini-Hochberg) instead: it allows some false positives among discoveries rather than controlling the chance of any false positive, giving more power when many tests are run.
When you see it
In subgroup analysis, multiple endpoint testing, feature selection, and any study design where pre-planned multiple comparisons are acknowledged and controlled.
Red flag
Ask whether the correction is appropriate for the decision. Bonferroni is often too conservative in exploratory screening — where discovering something is better than missing it. It is most appropriate when the cost of any single false positive is high.
Business example
A pharmaceutical company compares a new drug across 8 secondary endpoints alongside the primary endpoint. Using Bonferroni: each secondary test requires p < 0.05/8 = 0.00625 for significance. A secondary endpoint showing p = 0.02 would be non-significant — the correction is aggressive precisely because each false secondary claim has real regulatory and reputational consequences.
P-Hacking and HARKing
P-hacking: running analyses until significance is found. HARKing: Hypothesizing After Results are Known — both inflate false positive rates and undermine analytical credibility.
expand
Intuition
It tells you that analytical degrees of freedom — choices about which data to include, which metric to use, which subgroup to focus on — can be exploited to find significance that would not survive pre-registration. Most p-hacking is not intentional fraud; it is motivated reasoning applied to flexible analytical choices. The result is an epidemic of findings that fail to replicate.
Formula / Rule
Prevention: pre-register the primary hypothesis, metric, and analysis approach before data collection. Separate exploratory analysis (generate hypotheses) from confirmatory analysis (test them). Pre-registered hypotheses are protected from multiple testing inflation. Post-hoc hypotheses need replication in fresh data before being acted on.
When you see it
When experiment reports lead with subgroup findings rather than primary metrics, when the methodology section mentions "we also tested" extensively, or when conclusions change after the initial result was not significant.
Red flag
Ask whether the hypothesis was registered before the data was collected. If the most prominent finding in an A/B test report is a subgroup or secondary metric rather than the pre-specified primary outcome, treat it as an exploratory signal requiring replication.
Business example
A product team runs an A/B test on a checkout redesign. The primary metric (conversion rate) shows no significant improvement. The team then examines 15 subgroups and finds that mobile users aged 25–34 show significant improvement (p = 0.041). This subgroup analysis was not pre-specified — with 15 subgroups tested, approximately one false positive is expected. The finding should be treated as a hypothesis for a dedicated future test, not as evidence to justify a redesign.
Chapter Summary

Chapter 6 cheat sheet

Key principles from this chapter
p-value ≠ probability of null being true
p-value = P(data this extreme | H₀ true). Not P(H₀ true | this data). This distinction matters for every significance claim.
Statistical ≠ practical significance
With large samples, trivially small effects become significant. Always report effect size in business units alongside p-values.
Non-significant ≠ no effect
Failing to reject H₀ only means lack of evidence — not evidence of lack. An underpowered test is nearly uninterpretable.
Multiple tests require correction
Every test at α=0.05 has 5% false positive rate. 20 tests = ~64% chance of at least one false positive. Apply Bonferroni or FDR control.
Pre-register primary hypotheses
Pre-specified primary tests are protected from multiple testing inflation. Post-hoc subgroup findings are hypotheses requiring replication.
Power ≥ 80% before running tests
Calculate required sample size before collecting data. A test without 80% power to detect your minimum detectable effect probably shouldn't be run.
